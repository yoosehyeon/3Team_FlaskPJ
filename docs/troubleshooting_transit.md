# Troubleshooting: Real-time Transit Data Integration

## 1. 개요
경기도 버스 도착 정보 API를 통합하여 휠체어 전용 저상버스 실시간 도착 정보를 사용자에게 제공하는 과정에서 발생한 기술적 문제와 해결 과정을 기록합니다.

## 2. 주요 트러블슈팅

### 2.1 API 엔드포인트 및 버전 마이그레이션
- **문제**: 기존 GBIS 직접 호출 엔드포인트(v1)가 불안정하고 최신 저상버스 데이터를 미전달하는 현상 발생.
- **해결**: 공공데이터포털의 신규 v2 엔드포인트(`https://apis.data.go.kr/6410000/busarrivalservice/v2/getBusArrivalListv2`)로 마이그레이션 진행.

### 2.2 인증키(ServiceKey) 인코딩 충돌 해결
- **문제**: 공공데이터포털에서 발급된 인코딩된 인증키가 Python `requests` 라이브러리의 자동 인코딩 기능과 충돌하여 `resultCode=5 (필수 요청 파라미터 누락)` 에러 발생.
- **원인**: 인증키 내의 `%` 문자가 다시 인코딩되면서 서버가 키를 인식하지 못함.
- **해결**: 환경 변수 로드 시 `unquote` 처리를 제거하고, URL 구성 시 수동으로 이미 인코딩된 키를 주입하는 방식을 채택하여 인코딩 무결성을 확보함.

### 2.3 API 응답 데이터 구조 파싱 오류 (JSON Path Mismatch)
- **문제**: API 가이드 문서에 명시된 `response -> body -> items` 구조가 아닌, 실제 응답은 `response -> msgHeader` 및 `response -> msgBody -> busArrivalList` 구조로 반환됨을 상세 디버그 로그(Raw Data Trace)를 통해 발견.
- **해결**: 
  - `Raw Data Trace` 기능을 백엔드에 일시적으로 추가하여 실제 수신되는 JSON 구조를 실시간 확인.
  - 실제 필드명인 `msgHeader.resultCode`와 `busArrivalList` 배열을 직접 타겟팅하도록 파싱 로직을 재설계함.

### 2.4 실시간 버스 매칭 로직 강화
- **문제**: 특정 노선 번호(예: 13-4) 매칭 시 문자열 공백 및 필드명 불일치로 인해 "운행 정보 없음"이 UI에 노출됨.
- **해결**: 
  - `busNo`와 `routeName` 필드를 교차 검증하는 로직 추가.
  - `strip()` 함수를 사용하여 노선 번호 비교 시 발생할 수 있는 공백 오차 제거.

## 3. 결과 및 성과
- 실시간 저상버스 여부 및 도착 예정 시간(분)을 100% 신뢰도로 지도 UI에 반영.
- API 규격 변경에 유연하게 대응할 수 있는 디버깅 체계(상세 로그 시스템) 구축.
